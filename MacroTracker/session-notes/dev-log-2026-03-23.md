# Dev Log — 2026-03-23

## FatSecret Premier Tier Upgrade

Upgraded the FatSecret integration from basic to premier free tier after approval.

### What changed

**OAuth scope**: `basic` → `premier barcode`

**Structured nutrition via `food.get.v4`**
- Search now fetches full structured nutrition for each result instead of parsing the description string ("Per 100g - Calories: 195kcal | Fat: ...")
- Each food returns exact values for calories, carbs, protein, fat, fiber, sugar
- Multiple serving sizes returned as measures (e.g., "100g", "1/2 small breast", "1 oz boneless cooked")

**Barcode lookup implemented**
- `barcodeFatSecret()` was previously a stub returning null
- Now uses `food.find_id_for_barcode` to resolve barcode → food ID, then `food.get.v4` for full nutrition
- Acts as fallback when Open Food Facts doesn't have the product
- Barcode lookup chain: local DB cache → Open Food Facts → FatSecret

**Note**: Open Food Facts sometimes returns products with no useful data (e.g., "Unknown Product" with all-zero nutrition) which still counts as "found" and skips the FatSecret fallback. This could be tightened in a future session.

## Barcode Scanner Improvements

Improved camera scanning reliability for reading barcodes on physical products.

### What changed

- **Continuous autofocus** — After camera starts, applies `focusMode: 'continuous'` to the video track so the camera keeps hunting for focus instead of settling
- **1.5x zoom** — Slight zoom helps the autofocus system engage at typical barcode scanning distance
- **Higher FPS** — 10 → 15 scan attempts per second
- **Wider scan region** — Adapts to screen width (up to 300px) instead of fixed 250px
- **16:9 aspect ratio** — Requests higher-res camera mode for more detail
- **Flashlight toggle** — If device supports torch, a "Flashlight" button appears below the viewfinder

## TWA Chrome UI Investigation

The MacroTracker APK was showing Chrome browser elements (URL bar, X button, share icon) instead of running as a clean TWA like the Creighton app.

### Findings

- `assetlinks.json` is correctly served at `https://macros.stephens.page/.well-known/assetlinks.json`
- SHA-256 fingerprint in assetlinks.json matches the APK signing key
- Google's Digital Asset Links API confirms verification passes
- Server returns correct Content-Type (`application/json`), no redirects
- **Root cause**: Android/Chrome caches the asset link verification result. The APK was likely installed before the assetlinks.json was in place, and the cached failure persists.
- **Fix**: Clear Chrome cache + Android System WebView cache on the device, uninstall, and reinstall the same APK. No rebuild needed.
