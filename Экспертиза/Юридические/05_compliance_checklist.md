## 2. Соответствие 152-ФЗ (РФ)

### 2.1. Регистрация в качестве оператора ПДн
- ☐ Подано уведомление в Роскомнадзор о начале обработки персональных данных (rkn.gov.ru, форма онлайн)
- ☐ Получен реестровый номер оператора ПДн
- ☐ Номер указан в Политике конфиденциальности

### 2.2. Внутренние документы оператора (обязательны даже если не запрашивают)
- ☐ Приказ о назначении ответственного за организацию обработки ПДн
- ☐ Положение об обработке персональных данных
- ☐ Политика оператора в отношении обработки ПДн (отдельный документ от PP для пользователей)
- ☐ Перечень ПДн, обрабатываемых в организации
- ☐ Перечень мест хранения и обработки ПДн
- ☐ Журнал учёта обращений субъектов ПДн
- ☐ Порядок уничтожения ПДн (с актом уничтожения)
- ☐ Модель угроз безопасности ПДн (УБПДн) согласно методике ФСТЭК
- ☐ Перечень сотрудников, имеющих доступ к ПДн (для организаций с наёмным персоналом)

### 2.3. Согласия и формы
- ☐ Чек-бокс согласия при регистрации (НЕ pre-ticked)
- ☐ Отдельный чек-бокс на специальные категории ПДн (здоровье)
- ☐ Отдельный чек-бокс на трансграничную передачу
- ☐ Отдельный чек-бокс на маркетинговые коммуникации
- ☐ Таблица user_consents в БД с логами согласий

### 2.4. Трансграничная передача
- ☐ Подано **отдельное уведомление** в Роскомнадзор о трансграничной передаче (требование с 01.03.2023, 152-ФЗ в редакции 266-ФЗ)
- ☐ Получено подтверждение или истёк срок рассмотрения (10 рабочих дней)
- ☐ В Политике указан перечень стран и получателей

### 2.5. Локализация данных российских пользователей
- ☐ Первичный сбор / запись / систематизация / накопление / хранение / уточнение / извлечение данных РФ-граждан осуществляется на серверах на территории РФ (ст. 18 ч. 5 152-ФЗ)
- ⚠️ **Внимание:** Supabase размещена в ЕС. Для MVP допустимо запускаться на ЕС-инфраструктуре при корректном согласии на трансграничную передачу (позиция спорная, но практически применяется многими SaaS). **При масштабировании — рассмотреть локализацию на Yandex Cloud / Selectel / VK Cloud.**
- ☐ Подготовлен план миграции на РФ-инфраструктуру при достижении 5К+ активных РФ-пользователей

### 2.6. Права субъекта ПДн — техническая реализация
- ☐ Работает кнопка «Удалить аккаунт» в настройках
- ☐ Работает экспорт данных в JSON (переносимость данных, GDPR Art. 20)
- ☐ Процедура обработки запросов на privacy@gymfuel.ai (SLA 30 дней)
- ☐ Журнал обращений ведётся

## 3. Соответствие GDPR (ЕС)

### 3.1. До запуска на EU-рынке
- ☐ DPA (Data Processing Agreement) подписан с Supabase
- ☐ DPA подписан с Vercel
- ☐ DPA подписан с Railway
- ☐ DPA подписан с Google (Gemini API) — через Google Cloud Platform Terms
- ☐ DPA подписан с Anthropic (Claude API)
- ☐ DPA подписан со Stripe
- ☐ Подписаны Standard Contractual Clauses (SCC) с каждым US-подрядчиком
- ☐ Проведена TIA (Transfer Impact Assessment) для каждой US-передачи
- ☐ Подготовлен Privacy Policy EN с обязательными блоками GDPR (ст. 13–14)

### 3.2. DPO и EU Representative
- ☐ Оценён порог «регулярный мониторинг большого числа субъектов» — если достигнут → назначить DPO
- ☐ При регулярном обслуживании EU-резидентов — назначен EU Representative (ст. 27 GDPR)
- ☐ Контакты DPO / Representative опубликованы в Privacy Policy

### 3.3. DPIA (Data Protection Impact Assessment)
- ☐ Проведена DPIA для обработки специальных категорий ПДн (здоровье)
- ☐ DPIA задокументирована и хранится внутренне
- ☐ По результатам DPIA реализованы меры митигации

### 3.4. Реестр обработки (Article 30 GDPR)
- ☐ Ведётся внутренний реестр деятельности по обработке (ROPA — Records of Processing Activities)
- ☐ Реестр содержит: цели, категории субъектов, категории ПДн, получателей, сроки, меры безопасности

### 3.5. Cookie & ePrivacy
- ☐ Cookie-баннер реализован
- ☐ Согласие на cookie запрашивается ДО установки нессеншиал cookie
- ☐ Кнопки «Принять все» и «Отклонить все» одинакового размера (не тёмный паттерн)
- ☐ Настройки cookie сохраняются и применяются
- ☐ Cookie Policy опубликована

### 3.6. Breach Notification
- ☐ Разработана процедура уведомления о нарушениях безопасности ПДн
- ☐ SLA: уведомление в надзорный орган — в течение 72 часов
- ☐ Уведомление субъектов — без необоснованной задержки (если высокий риск)
- ☐ Назначен ответственный за реагирование на инциденты

## 7. Безопасность (технические требования)

### 7.1. Базовая защита
- ☐ HTTPS везде (HSTS включён)
- ☐ TLS 1.2+ (TLS 1.0/1.1 отключены)
- ☐ CSP (Content Security Policy) заголовки настроены
- ☐ X-Frame-Options / clickjacking защита
- ☐ Пароли хешируются bcrypt (cost ≥ 12)
- ☐ JWT с refresh tokens и коротким TTL access
- ☐ Rate limiting на login и registration endpoints
- ☐ CAPTCHA (reCAPTCHA / hCaptcha) на публичных формах

### 7.2. База данных
- ☐ Row-Level Security включён на всех таблицах с user_id (Supabase RLS)
- ☐ Политики RLS проверены: `auth.uid() = user_id`
- ☐ Роли БД разграничены (anon, authenticated, service_role)
- ☐ Бэкапы ежедневные, хранение 90 дней
- ☐ Test restore процедура выполнена (хотя бы раз)

### 7.3. Файловое хранилище
- ☐ Приватный bucket в Supabase Storage (не public)
- ☐ Signed URLs с коротким TTL для доступа к фото
- ☐ Шифрование at-rest включено (Supabase по умолчанию)

### 7.4. Секреты и API-ключи
- ☐ Секреты НЕ в коде / git — только в переменных окружения
- ☐ Разные ключи для dev / staging / production
- ☐ Gemini API / Claude API ключи проксируются через backend (не экспонируются на frontend)
- ☐ .env файлы в .gitignore
- ☐ Процедура ротации ключей описана (раз в 90 дней)

### 7.5. Мониторинг и логирование
- ☐ Настроен мониторинг uptime (UptimeRobot / Better Stack)
- ☐ Error tracking (Sentry / Rollbar)
- ☐ Логи не содержат ПДн в открытом виде
- ☐ Alerts настроены на критичные события (массовые 500-ки, failed logins, DB downtime)

### 7.6. Аудит безопасности
- ☐ Pre-launch чеклист OWASP ASVS Level 1 пройден
- ☐ Статический анализ кода (Semgrep / Snyk) запущен
- ☐ Проверка зависимостей на уязвимости (npm audit / pip-audit)
- ☐ Проведён хотя бы внутренний security review
